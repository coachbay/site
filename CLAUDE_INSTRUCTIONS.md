## PDF Standard

CoachBay uses `@react-pdf/renderer` for dynamic PDFs (Disruption Sprint) and `jsPDF` for diagnostic PDFs (DiagnosticEngine.jsx). Both are valid depending on the use case.

### Font Rules
Use built-in Helvetica only in react-pdf. Custom woff2/ttf fonts cause glyph subsetting errors with dynamic content. Register a hyphenation callback to prevent unwanted word breaks:
```js
Font.registerHyphenationCallback(word => [word]);
```

### Colour Constants
```js
const C = {
  cyan:       "#00BCD4",
  cyanDark:   "#0097A7",
  cyanLight:  "#E0F7FA",
  cyanBorder: "#80DEEA",
  navy:       "#1A1A2E",
  navyText:   "#1E293B",
  body:       "#334155",
  muted:      "#64748B",
  divider:    "#E2E8F0",
  bg:         "#F8FAFC",
  white:      "#FFFFFF",
  threatHigh: "#7F1D1D",
  threatMid:  "#92400E",
  threatLow:  "#166534",
};
```

### Header and Footer Pattern
- **Header**: White bar with robot PNG + "CoachBay.ai" left, document label right, light bottom border
- **Footer**: Left side — "coachbay.ai · Generated by CoachBay · [date]". Right side — dynamic page number using `render` prop
- **Footer uses `fixed` prop** (safe, since it uses absolute positioning)
- **Header must NOT use `fixed` prop** — see architecture rules below

### react-pdf Architecture Rules (CRITICAL)

These rules were learned in April 2026. Breaking them causes layout bugs that are very hard to debug.

1. **NEVER use `fixed` on headers.** The `fixed` prop makes the header repeat on every page, but it causes `paddingTop` on the Page to be ignored on overflow pages. Content ends up jammed against the header with no breathing room. No amount of padding fixes this.

2. **Split long documents into separate `<Page>` components per section.** Instead of one big `<Page wrap>`, create a separate `<Page>` for each major section. Each section gets its own HeaderBand and Footer. This way every section starts on a fresh page with proper spacing.

3. **Always use `wrap={false}` on cards.** Wrap every card in `<View wrap={false}>` so it moves to the next page as a whole unit rather than being sliced in half at a page break.

4. **Footer can remain `fixed`.** It uses `position: "absolute"` with `bottom: 14`, so it does not interfere with content flow. Use the `render` prop for dynamic page numbers:
```jsx
<Text render={({ pageNumber, totalPages }) => `Page ${pageNumber} of ${totalPages}`} />
```

### Canonical Reference
`DisruptionSprintPDF.jsx` is the canonical pattern for all future CoachBay PDFs. Use it as the template when building new PDF components.

### Browser Integration
In React components, use `pdf()` from `@react-pdf/renderer` to generate a blob, then trigger a download via a temporary anchor element. See the download handler in `DisruptionSprint.jsx` for the pattern.

## Section Icons (DiagnosticEngine.jsx)

All three diagnostics use flat cyan SVG line icons from Lucide (or custom). Icons are defined in the `SECTION_ICONS` map at the top of `DiagnosticEngine.jsx`. The key in `diagnosticData.js` maps to an icon here.

| Key | Icon | Used for |
|---|---|---|
| `target` | Target/bullseye | Strategic Clarity / Integration |
| `leader` | Compass | Leadership Readiness |
| `chat` | Chat bubble | Employee Sentiment |
| `cycle` | Cycle arrows | Culture of Change |
| `tools` | Wrench (Lucide) | Practical Foundations |
| `brain` | Layers (Lucide) | Depth |
| `scissors` | Scissors | Deletion |
| `scales` | Scales | Judgment |
| `handshake` | Users/two people (Lucide) | Enabling / Influence |
| `leaf` | Leaf (Lucide) | Environment |

**Rules:**
- Always use flat cyan stroke icons, `stroke="#00BCD4"`, no fill
- Use Lucide icon paths where possible — they are designed to render cleanly at small sizes
- Never use emoji as section icons

## Client Dashboard URLs — Critical Rule

Every slug in `clientConfig.js` automatically has a live dashboard. The URL is always:
  coachbay.ai/[slug]-dashboard

Examples:
  cpcs-sym     → coachbay.ai/cpcs-sym-dashboard
  scchk-core   → coachbay.ai/scchk-core-dashboard
  finnair-sym  → coachbay.ai/finnair-sym-dashboard

Before telling Tomas a dashboard does not exist or has not been built, always read
clientConfig.js and derive the URL from the slug. The LiveDashboard.jsx component
and dynamic routing in App.jsx handle all slugs automatically.

The automation that runs for every client:
- Dashboard loads live data from the Google Sheet on every page visit
- AI generates fresh insights each time
- Cron job fires at 9am HKT on the closeDate and emails Tomas with submission
  count, dashboard link, and a draft client email ready to forward
